## Structure

```
apps/
  console/                 # Sandbox Dashboard (Next.js)
  ui-docs/                 # UI component docs (Vite)
packages/
  cli/                     # TypeScript CLI (@superserve/cli)
  python-sdk/              # Python SDK (superserve on PyPI)
  sdk/                     # TypeScript SDK (@superserve/sdk on npm)
  ui/                      # Shared UI components
  supabase/                # Supabase client factories
  typescript-config/       # Shared tsconfig presets
  tailwind-config/         # Shared Tailwind config
  biome-config/            # Shared Biome config
fern/                      # Fern SDK + docs generation config
  generators.yml           # TS + Python generator config
  overlay-files.yaml       # Data-plane file endpoints (overlay)
  docs.yml                 # Fern Docs site config
  fonts/                   # Vendored WOFF2 fonts for docs
  styles.css               # Custom CSS for docs
docs/
  pages/                   # MDX content for Fern Docs
  logo/                    # Logo assets
tests/
  sdk-e2e-ts/              # TypeScript SDK e2e tests (Vitest)
  sdk-e2e-py/              # Python SDK e2e tests (pytest)
spec/                      # Planning and design documents
```

## Development

This repo is a monorepo managed with [Bun workspaces](https://bun.sh/docs/install/workspaces), [Turborepo](https://turbo.build/repo), and [uv workspaces](https://docs.astral.sh/uv/concepts/projects/workspaces/) (for Python).

### Setup

```bash
bun install               # install all JS/TS dependencies
uv sync                   # install all Python dependencies
```

### Day-to-day commands

Run tasks across all packages from the repo root:

```bash
bun run build             # build all packages
bun run dev               # start all dev servers
bun run lint              # lint all packages
bun run typecheck         # type check all packages
bun run test              # run all unit/integration tests (fast, no credentials needed)
```

Target a specific package with `--filter`:

```bash
bunx turbo run dev --filter=@superserve/console
bunx turbo run build --filter=@superserve/sdk
```

### Adding dependencies

Always add dependencies from the repo root using `--filter`:

```bash
bun add zod --filter @superserve/cli
bun add react --filter @superserve/console
bun add -d @types/node --filter @superserve/sdk
```

> **Note:** Do not `cd` into a package directory and run `bun add` directly — this creates a separate `bun.lock` that conflicts with the monorepo's root lockfile.

## SDKs & Docs

The Superserve SDKs (`@superserve/sdk`, `superserve`) and the [docs site](https://superserve.docs.buildwithfern.com) are generated by [Fern](https://buildwithfern.com) from the OpenAPI spec in the backend repo ([`github.com/superserve-ai/sandbox/api/openapi.yaml`](https://github.com/superserve-ai/sandbox)). This monorepo references that URL directly — the spec is fetched on demand into `fern/openapi.yaml` (gitignored) every time you run a Fern command. There is no committed copy.

### Regenerating the SDKs

Run this when the backend team has merged a spec change and you want the SDKs updated:

```bash
bun run generate           # fetches latest spec from backend repo, regenerates TS + Python sources
bun run build              # rebuilds dist (tsup for TS, hatchling for Python)
```

Generation runs on Fern's cloud — no Docker required. To validate the spec without generating:

```bash
bun run generate:check     # fetches spec + runs fern check (lint/validation only)
```

If you prefer CI, trigger the **Fern — Regenerate SDKs** workflow manually in GitHub Actions; it will open a PR with the regenerated files.

### Previewing the docs locally

```bash
bun run docs:dev           # fetches spec + starts a local Fern Docs dev server
```

Opens a local preview (typically `localhost:3000`) with the full docs site including the API Reference, quickstarts, and SDK code samples. Hot-reloads on MDX changes.

### Publishing the docs site

```bash
bun run docs:publish       # fetches spec + publishes to superserve.docs.buildwithfern.com
```

Or trigger the **Publish Fern Docs** workflow in GitHub Actions. Requires `FERN_TOKEN`.

### Publishing the TypeScript SDK to npm

From the repo root:

```bash
cd packages/sdk
# bump `version` in package.json
bun run build
NPM_CONFIG_TOKEN=npm_... bun publish --access public
```

Or with `.env.release` sourced (see [Release environment setup](#release-environment-setup) below):

```bash
source .env.release
cd packages/sdk
bun run build
bun publish --access public
```

### Publishing the Python SDK to PyPI

From the **repo root** (not from `packages/python-sdk/` — uv workspaces output build artifacts to the repo root `dist/`):

```bash
uv build --package superserve
UV_PUBLISH_TOKEN=pypi-... uv publish dist/superserve-<version>*
```

Or with `.env.release` sourced:

```bash
source .env.release
uv build --package superserve
uv publish dist/superserve-<version>*
```

> **Important:** `uv build` must be run from the repo root with `--package superserve`. Running `uv build` from inside `packages/python-sdk/` puts the output in the repo root `dist/` directory, and `uv publish` from the package directory won't find them.

### Release environment setup

For manual releases from your laptop, copy `.env.release.example` to `.env.release`, fill in the tokens, and `source` it before running any publish command. `.env.release` is gitignored.

```bash
cp .env.release.example .env.release
# edit .env.release and fill in the three tokens
source .env.release
```

The three tokens you need:

| Env variable | Used by | How to get it |
|---|---|---|
| `FERN_TOKEN` | `bun run docs:publish` | [dashboard.buildwithfern.com](https://dashboard.buildwithfern.com) → settings → API tokens |
| `NPM_CONFIG_TOKEN` | `bun publish` (TS SDK) | [npmjs.com](https://www.npmjs.com) → account settings → access tokens → generate automation token |
| `UV_PUBLISH_TOKEN` | `uv publish` (Python SDK) | [pypi.org](https://pypi.org/manage/account/) → account settings → API tokens → add token (starts with `pypi-`) |

### CI workflows (GitHub Actions)

All three workflows are `workflow_dispatch` only (manual trigger from the Actions UI):

| Workflow | What it does |
|---|---|
| **Fern — Regenerate SDKs** | Fetches latest spec, runs `bun run generate`, opens a PR with the regenerated files |
| **Release SDKs** | Bumps version, builds, publishes to npm and/or PyPI, commits + tags. Inputs: `package` (`ts` / `python` / `both`), `version` |
| **Publish Fern Docs** | Fetches spec, publishes docs to `superserve.docs.buildwithfern.com` |

Required repo secrets (one-time setup in GitHub repo settings → Secrets and variables → Actions):

| Secret | Workflow |
|---|---|
| `NPM_TOKEN` | Release SDKs (ts) |
| `PYPI_TOKEN` | Release SDKs (python) |
| `FERN_TOKEN` | Publish Fern Docs |

## Testing

### Unit / integration tests (fast, no credentials)

```bash
bun run test               # runs all unit tests via turbo (console, cli, sdk)
```

### End-to-end tests (credentialed, hits live API)

```bash
SUPERSERVE_API_KEY=ss_live_... bun run test:e2e
```

Runs both the TypeScript (Vitest) and Python (pytest) e2e suites against a live Superserve environment. Defaults to staging; override with `SUPERSERVE_BASE_URL`:

```bash
# against production
SUPERSERVE_API_KEY=ss_live_... SUPERSERVE_BASE_URL=https://api.superserve.ai bun run test:e2e

# against local dev
SUPERSERVE_API_KEY=ss_dev_... SUPERSERVE_BASE_URL=http://localhost:8080 bun run test:e2e
```

Run just one language:

```bash
SUPERSERVE_API_KEY=ss_live_... bunx turbo run e2e --filter=@superserve/test-sdk-e2e-ts
SUPERSERVE_API_KEY=ss_live_... bunx turbo run e2e --filter=@superserve/test-sdk-e2e-py
```

Without `SUPERSERVE_API_KEY`, all e2e tests skip cleanly (exit 0). The e2e suites are **not** part of `bun run test` — they only run when explicitly invoked via `bun run test:e2e`.

### Coverage

| Suite | Test files | What it covers |
|---|---|---|
| TS e2e | `tests/sdk-e2e-ts/tests/{system,sandboxes,exec,files}.test.ts` | health, create/get/list/patch/pause/resume/delete sandboxes, sync + streaming exec, file upload/download |
| Python e2e | `tests/sdk-e2e-py/test_{system,sandboxes,exec,files,async}.py` | same as TS, plus async client smoke test |
| Console unit | `apps/console/src/**/*.test.{ts,tsx}` | auth flows, components, API proxy |
| CLI unit | `packages/cli/tests/**` | config validation, deploy logic |
