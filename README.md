## Development

This repo is a monorepo managed with [Bun workspaces](https://bun.sh/docs/install/workspaces) and [Turborepo](https://turbo.build/repo).

### Setup

```bash
bun install               # install all dependencies
```

### Running Commands

Run tasks across all packages from the repo root:

```bash
bun run build             # build all packages
bun run dev               # start all dev servers
bun run lint              # lint all packages
bun run typecheck         # type check all packages
bun run test              # run all tests
```

Target a specific package with `--filter`:

```bash
bunx turbo run dev --filter=@superserve/console
bunx turbo run build --filter=@superserve/sdk
```

### Adding Dependencies

Always add dependencies from the repo root using `--filter`:

```bash
bun add zod --filter @superserve/cli
bun add react --filter @superserve/console
bun add -d @types/node --filter @superserve/sdk
```

> **Note:** Do not `cd` into a package directory and run `bun add` directly - this creates a separate `bun.lock` that conflicts with the monorepo's root lockfile.

### Python SDK

The Python SDK lives in `packages/python-sdk/` and is managed with **uv workspaces**:

```bash
uv sync                                              # install Python dependencies
uv run pytest packages/python-sdk/tests/              # run tests
uv run ruff check packages/python-sdk/                # lint
uv run mypy packages/python-sdk/src/superserve/       # type check
```

## SDKs & Docs

The Superserve SDKs (`@superserve/sdk`, `superserve`) and the docs site are generated by [Fern](https://buildwithfern.com) from the OpenAPI spec in the backend repo (`github.com/superserve-ai/sandbox/api/openapi.yaml`). This monorepo references that URL directly — there is no committed copy of the spec. The spec is fetched on demand into `fern/openapi.yaml` (gitignored) every time you run a Fern command.

### Regenerating the SDKs

Run this when the backend team has merged a spec change and you want the SDKs updated:

```bash
bun run generate       # fetches latest spec, regenerates TS + Python sources
bun run build          # tsup TS, uv build Python
```

Requires Docker to be running (Fern runs generators in Docker containers).

If you prefer CI, trigger the **Fern — Regenerate SDKs** workflow manually in GitHub Actions; it will open a PR with the regenerated files.

### Publishing to npm and PyPI

**Manually from your laptop:**

```bash
# TypeScript
cd packages/sdk
# bump `version` in package.json
bun run build
bun publish --access public

# Python
cd packages/python-sdk
# bump `version` in pyproject.toml
uv build
uv publish             # requires UV_PUBLISH_TOKEN
```

**From CI:** trigger the **Release SDKs** workflow in GitHub Actions with `package` (`ts` / `python` / `both`) and `version` inputs. Requires `NPM_TOKEN` and `PYPI_TOKEN` repo secrets.

### Publishing the docs site

```bash
bun run docs:publish   # syncs spec, then fern generate --docs
```

Or trigger the **Publish Fern Docs** workflow in GitHub Actions. Requires `FERN_TOKEN` repo secret.

To preview the docs locally:

```bash
bun run docs:dev
```

### Release secrets

For CI releases, configure these as GitHub Actions repo secrets:

| Secret        | Used by                 | Source                                                       |
|---------------|-------------------------|--------------------------------------------------------------|
| `NPM_TOKEN`   | Release SDKs (ts)       | `npmjs.com` → account settings → access tokens → automation  |
| `PYPI_TOKEN`  | Release SDKs (python)   | `pypi.org` → account settings → API tokens                   |
| `FERN_TOKEN`  | Publish Fern Docs       | `dashboard.buildwithfern.com` → settings → API tokens        |

For **manual releases from your laptop**, copy `.env.release.example` to `.env.release`, fill in the tokens, and `source .env.release` before running any publish command. `.env.release` is gitignored.
