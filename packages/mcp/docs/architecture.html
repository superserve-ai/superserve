@superserve/mcp — Architecture

A stdio Model Context Protocol server that exposes 10 fine-grained tools over JSON-RPC, run via npx -y @superserve/mcp. It is a thin adapter that wraps @superserve/sdk and targets a sandbox per call by id — the per-sandbox data-plane access token never leaves the SDK.

Request path

  • 1. Host spawns npx -y @superserve/mcp with the API key in env
  • 2. Model emits a tool call → JSON-RPC over stdio
  • 3. McpServer validates input with zod, dispatches to the tool
  • 4. Tool calls SandboxClient, which connect(id)s a fresh SDK session
  • 5. SDK hits the control (API key) or data (access token) plane
  • 6. Result returns as human text + machine structuredContent, truncated

10 tools · 3 modules

  • Lifecycle (6): create, list, info, pause, resume, kill
  • Exec (1): sandbox_exec — auto-resumes a paused VM first
  • Files (3): read, write, list (text / base64)
  • info() is read-only — resolved via list so it never resumes
  • files_list shims the missing SDK op: find -printf → ls fallback
  • Honest annotations (readOnly / destructive / idempotent) guide client UX

Trust & credentials

  • Server holds only the control-plane API key (SUPERSERVE_API_KEY)
  • The data-plane access token stays inside the SDK — never over stdio, never to the model
  • Token is rotated on resume and re-injected by the SDK transparently
  • Per-call connect(id) = fresh token, no cross-call session cache
  • Errors mapped to safe, actionable strings — no credential leakage
  • All diagnostics go to stderr; stdout is reserved for the protocol
