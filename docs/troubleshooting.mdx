## VM stuck in CREATING

**Cause:** The VM provisioning process is taking longer than expected, or the image is not available.

**Fix:** Check the VM status and delete it if it doesn't transition to `RUNNING` within a few seconds:

```bash
superserve vm get <vm_id>
superserve vm delete <vm_id> --force
```

Then create a new VM. If the issue persists, verify that the image name is correct (e.g. `ubuntu-22.04`).

---

## Command execution times out

**Cause:** The command exceeds the default timeout, or the VM is in a `STOPPED` or `SLEEPING` state.

**Fix:**

1. Check the VM status:

```bash
superserve vm get <vm_id>
```

2. If the VM is stopped or sleeping, start or wake it first:

```bash
superserve vm start <vm_id>
# or
superserve vm wake <vm_id>
```

3. Increase the timeout for long-running commands:

```bash
superserve exec <vm_id> "python3 /app/train.py" --timeout 600
```

---

## File upload fails

**Cause:** The remote path doesn't exist, or the VM is not running.

**Fix:**

1. Make sure the VM is in `RUNNING` state.
2. Ensure the parent directory exists on the VM:

```bash
superserve exec <vm_id> "mkdir -p /app"
superserve files upload <vm_id> ./script.py /app/script.py
```

## Rollback doesn't restore expected state

**Cause:** The checkpoint was created at a different point than expected, or the wrong checkpoint was specified.

**Fix:** List checkpoints to see what's available and verify timestamps:

```bash
superserve checkpoint list <vm_id>
```

Use `--name` for clarity when creating checkpoints:

```bash
superserve checkpoint create <vm_id> --name before-experiment
```

<Tip>
  Create checkpoints at well-defined points in your workflow. Name them descriptively so you can identify them later.
</Tip>

## Fork produces VMs that diverge unexpectedly

**Cause:** Forks share state at the moment of forking but are fully independent afterward. If background processes are running in the original VM, each fork gets its own copy of those processes, which may behave differently.

**Fix:** For reproducible forks, stop any background processes or create a checkpoint from a quiescent state before forking:

```bash
superserve exec <vm_id> "pkill -f background-process"
superserve checkpoint create <vm_id> --name clean-state
superserve fork <vm_id> --from-checkpoint <checkpoint_id> --count 3
```

## Authentication fails

**Cause:** Missing or invalid API key.

**Fix:**

1. Check that your API key is set:

```bash
echo $SUPERSERVE_API_KEY
```

2. Re-authenticate with the CLI:

```bash
superserve login --api-key ss_your_api_key
```

3. Generate a new API key from the [Superserve Console](https://console.superserve.ai) if needed.

---

## SSH connection refused

**Cause:** The VM is not in `RUNNING` state, or it hasn't finished booting.

**Fix:**

```bash
superserve vm get <vm_id>
```

If the status is not `RUNNING`, start or wake the VM first. If it just transitioned to `RUNNING`, wait a moment for the SSH daemon to initialize.
