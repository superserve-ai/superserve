---
title: "Checkpoints"
description: "Checkpoint, rollback, and fork concepts"
---

Checkpoints capture the full state of a VM -- filesystem, memory, and running processes. Use them to roll back destructive changes, branch execution, or create reproducible starting points.

## Creating a checkpoint

A checkpoint snapshots the VM at a point in time. The VM continues running after the checkpoint is created.

<CodeGroup>

```bash CLI
superserve checkpoint create vm_a1b2c3d4 --name baseline
```

```typescript TypeScript
const cp = await client.checkpoints.create("vm_a1b2c3d4", {
  name: "baseline",
})
```

```python Python
cp = client.checkpoints.create("vm_a1b2c3d4", name="baseline")
```

</CodeGroup>

You can create multiple checkpoints for a single VM. Each one is independent and can be used for rollback or forking.


```typescript TypeScript
const checkpoints = await client.checkpoints.list("vm_a1b2c3d4")
```

```python Python

```

</CodeGroup>

## Rollback

Restore a VM to a previous checkpoint. The VM's filesystem, memory, and processes are reverted to the exact state when the checkpoint was created. Everything that happened after the checkpoint is discarded.

<CodeGroup>

```bash CLI
# By name
superserve rollback vm_a1b2c3d4 --name baseline

# By checkpoint ID
superserve rollback vm_a1b2c3d4 --checkpoint cp_x7y8z9

# By time
superserve rollback vm_a1b2c3d4 --minutes-ago 30
```

```typescript TypeScript
// By checkpoint ID
await client.rollback("vm_a1b2c3d4", { checkpointId: "cp_x7y8z9" })

// By name
await client.rollback("vm_a1b2c3d4", { name: "baseline" })

// By time
await client.rollback("vm_a1b2c3d4", { minutesAgo: 30 })
```

```python Python
# By checkpoint ID
client.rollback("vm_a1b2c3d4", checkpoint_id="cp_x7y8z9")

# By name
client.rollback("vm_a1b2c3d4", name="baseline")

# By time
client.rollback("vm_a1b2c3d4", minutes_ago=30)
```

</CodeGroup>

<Warning>
  Rollback is destructive. All changes made after the checkpoint are lost. Create a new checkpoint before rolling back if you want to preserve the current state.
</Warning>

## Fork

Fork creates one or more independent copies of a VM. Each fork is a fully separate VM with its own ID, but starts with the exact same state as the original.

<CodeGroup>

```bash CLI
superserve fork vm_a1b2c3d4 --count 3
```

```typescript TypeScript
const forks = await client.fork("vm_a1b2c3d4", { count: 3 })
```

```python Python
forks = client.fork("vm_a1b2c3d4", count=3)
```

</CodeGroup>

### Fork from a checkpoint

Fork from a specific checkpoint instead of the current state. This is useful for creating multiple independent runs from a known-good starting point.

<CodeGroup>

```bash CLI
superserve fork vm_a1b2c3d4 --from-checkpoint cp_x7y8z9 --count 5
```

```typescript TypeScript
const forks = await client.fork("vm_a1b2c3d4", {
  count: 5,
  fromCheckpoint: "cp_x7y8z9",
})
```

```python Python
forks = client.fork("vm_a1b2c3d4", count=5, from_checkpoint="cp_x7y8z9")
```

</CodeGroup>

### Fork tree

View the parent-child relationships between a VM and its forks.

```bash
superserve fork tree vm_a1b2c3d4
```

```
vm_a1b2c3d4 (original)
  ├── vm_fork_001
  ├── vm_fork_002
  └── vm_fork_003
        ├── vm_fork_004
        └── vm_fork_005
```

Forks can be forked again, creating a tree of independent VMs.

## Patterns

### Checkpoint before risky operations

```python
# Save state before running untrusted code
cp = client.checkpoints.create(vm.id, name="pre-experiment")

result = client.exec(vm.id, command="python3 /app/untrusted_script.py")

if result.exit_code != 0:
    # Something went wrong, roll back
    client.rollback(vm.id, checkpoint_id=cp.id)
```

### Parallel experiments with fork

```python
# Set up a base environment
vm = client.vms.create(name="base", image="ubuntu-22.04")
client.exec(vm.id, command="apt-get update && apt-get install -y python3 python3-pip")
client.files.upload(vm.id, "/app/train.py", training_script)
client.checkpoints.create(vm.id, name="ready")

# Fork for parallel experiments
forks = client.fork(vm.id, count=3)

configs = ["config_a.json", "config_b.json", "config_c.json"]
for fork, config in zip(forks, configs):
    client.files.upload(fork.id, "/app/config.json", config)
    client.exec(fork.id, command="python3 /app/train.py")
```

### Iterative rollback

```python
# Try multiple approaches, rolling back between each
cp = client.checkpoints.create(vm.id, name="starting-point")

for approach in approaches:
    client.files.upload(vm.id, "/app/solution.py", approach)
    result = client.exec(vm.id, command="python3 /app/test.py")

    if result.exit_code == 0:
        break  # Tests pass, keep this state

    # Tests failed, roll back and try the next approach
    client.rollback(vm.id, checkpoint_id=cp.id)
```

## Deleting checkpoints

<CodeGroup>

```bash CLI
superserve checkpoint delete vm_a1b2c3d4 cp_x7y8z9
superserve checkpoint delete vm_a1b2c3d4 cp_x7y8z9 --force
```

```typescript TypeScript
await client.checkpoints.delete("vm_a1b2c3d4", "cp_x7y8z9")
```

```python Python
client.checkpoints.delete("vm_a1b2c3d4", "cp_x7y8z9")
```

</CodeGroup>

Deleting a checkpoint does not affect the VM or any forks created from that checkpoint.
