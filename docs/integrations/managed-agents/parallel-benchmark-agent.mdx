A Claude Managed Agent that benchmarks code across multiple configurations simultaneously. The agent designs the variant matrix, writes a parallel harness that spins up N isolated Superserve sandboxes — one per variant — runs them all concurrently, collects timing results, and synthesizes a comparison report.

**The core pitch:** Benchmarking 9 configurations sequentially takes 9× longer than benchmarking them in parallel. Superserve sandboxes boot in under a second and run truly isolated. The agent drives all of this autonomously.

<Tip>
A [reference implementation](https://github.com/superserve-ai/superserve/tree/main/guides/claude/parallel-benchmark-agent) with the orchestrator, runner, and setup scripts is available on GitHub in both Python and TypeScript.
</Tip>

## Why this is hard to replicate elsewhere

**True isolation per variant.** A single sandbox with 4 vCPUs can parallelize across threads, but variants share CPU, memory, and OS state — results are noisy. Superserve gives you a separate VM, separate kernel, separate filesystem per variant. You can test "clean environment with only numpy 1.x" vs. "clean environment with numpy 2.x" — that is not possible with threads.

**Agent-driven parallelism.** The agent does not just run a benchmark — it *designs* the variant matrix, writes the parallel harness, manages the sub-sandbox lifecycle, and synthesizes the results. The user describes what to compare; the agent figures out how.

**Timing comparison.** If each variant takes 30 seconds, 9 variants = 4.5 minutes sequentially. With parallel sandboxes: still 30 seconds total.

## How it works

```
User: "Benchmark bubble sort, quicksort, and Python's built-in sort
       on 10k, 100k, and 1M random integers. 3 trials each."

Agent reasoning:
  → 3 algorithms × 3 sizes = 9 variants
  → writes /workspace/benchmark.py
  → writes /workspace/run_parallel.py
  → runs: SUPERSERVE_API_KEY=$SUPERSERVE_API_KEY python3 /workspace/run_parallel.py

run_parallel.py (agent-authored):
  → asyncio.gather(create sandbox × 9)     ← 9 sandboxes boot in ~1s each
  → asyncio.gather(upload + run × 9)       ← all 9 run simultaneously
  → asyncio.gather(collect results × 9)
  → asyncio.gather(kill × 9)
  → writes /workspace/results.md

Agent reads results.md, synthesizes:
  → "Python's built-in sort (Timsort) is 40× faster than bubble sort
     at 1M elements. Quicksort is 15× faster. Here's the full table..."
```

The agent-authored harness looks like this:

```python
import asyncio, os
from superserve import AsyncSandbox

VARIANTS = [
    ("bubble_sort",  10_000), ("bubble_sort",  100_000), ("bubble_sort",  1_000_000),
    ("quicksort",    10_000), ("quicksort",    100_000), ("quicksort",    1_000_000),
    ("builtin_sort", 10_000), ("builtin_sort", 100_000), ("builtin_sort", 1_000_000),
]

async def run_variant(algo, size):
    sandbox = await AsyncSandbox.create(name=f"bench-{algo}-{size}")
    await sandbox.files.write("/bench.py", open("/workspace/benchmark.py").read())
    result = await sandbox.commands.run(f"python3 /bench.py {algo} {size}")
    await sandbox.kill()
    return {"algo": algo, "size": size, "output": result.stdout}

results = asyncio.run(asyncio.gather(*[run_variant(a, s) for a, s in VARIANTS]))
```

## Build the sandbox template

The benchmark agent template installs the `superserve` Python SDK inside the sandbox. The agent uses it from `bash` to create sub-sandboxes. This is what makes the hierarchical pattern work.

<CodeGroup>
```typescript TypeScript
import { Template } from "@superserve/sdk"

const template = await Template.create({
  name: "claude-benchmark-agent",
  from: "python:3.12-slim",
  vcpu: 2,
  memoryMib: 2048,
  steps: [
    {
      run:
        "apt-get update && apt-get install -y --no-install-recommends " +
        "curl git jq procps && rm -rf /var/lib/apt/lists/*",
    },
    { run: "pip install --no-cache-dir anthropic superserve" },
    { run: "mkdir -p /workspace /mnt/session/outputs" },
    { workdir: "/workspace" },
  ],
})

await template.waitUntilReady({
  onLog: (ev) => {
    if (ev.stream !== "system") process.stdout.write(ev.text)
  },
})
```

```python Python
from superserve import Template, RunStep, WorkdirStep

template = Template.create(
    name="claude-benchmark-agent",
    from_="python:3.12-slim",
    vcpu=2,
    memory_mib=2048,
    steps=[
        RunStep(run=(
            "apt-get update && apt-get install -y --no-install-recommends "
            "curl git jq procps && rm -rf /var/lib/apt/lists/*"
        )),
        RunStep(run="pip install --no-cache-dir anthropic superserve"),
        RunStep(run="mkdir -p /workspace /mnt/session/outputs"),
        WorkdirStep(workdir="/workspace"),
    ],
)

template.wait_until_ready(
    on_log=lambda ev: print(ev.text, end="")
    if ev.stream.value != "system"
    else None,
)
```
</CodeGroup>

## Create the agent

The agent uses only `bash`, `read`, and `write` — benchmarks are self-contained and do not need web access. Its system prompt instructs it to design a variant matrix, write both scripts, run them, and synthesize the results.

<CodeGroup>
```typescript TypeScript
const agent = await client.beta.agents.create({
  name: "benchmark-agent",
  model: "claude-sonnet-4-6",
  system: `You are a benchmarking agent. The Superserve Python SDK is installed in your sandbox.

When asked to benchmark code:
1. Design the variant matrix (algorithm × parameter × size)
2. Write the benchmark script at /workspace/benchmark.py
3. Write a parallel harness at /workspace/run_parallel.py that:
   - Uses asyncio + AsyncSandbox to create one sandbox per variant
   - Uploads benchmark.py to each, runs it, collects stdout
   - Kills all sub-sandboxes when done
   - Writes a results table to /workspace/results.md
4. Run: SUPERSERVE_API_KEY=$SUPERSERVE_API_KEY python3 /workspace/run_parallel.py
5. Read results.md and report the comparison with a clear winner

Keep variant count <= 20. Always kill sub-sandboxes — never leave them running.
Sub-sandboxes should use from_template=None (platform default) unless the benchmark
requires specific packages.`,
  tools: [{
    type: "agent_toolset_20260401",
    default_config: { enabled: false, permission_policy: { type: "always_allow" } },
    configs: ["bash", "read", "write"].map((t) => ({
      name: t,
      enabled: true,
      permission_policy: { type: "always_allow" },
    })),
  }],
})
```


## Run the orchestrator

The benchmark agent orchestrator differs from the base in two ways:

1. **Network config** — allow `api.superserve.ai` so the agent can call the Superserve API to create sub-sandboxes:

<CodeGroup>
```typescript TypeScript
network: { allowOut: ["api.anthropic.com", "api.superserve.ai"] }
```

```python Python

```
</CodeGroup>

2. **Runner environment** — pass `SUPERSERVE_API_KEY` so the agent's sub-sandbox calls authenticate correctly:

## Sub-sandbox billing

Sub-sandboxes accrue charges while running. The agent-written harness always kills them immediately after collecting results. A typical benchmark run: 9 sandboxes × 30 seconds each = 4.5 sandbox-minutes. The harness guarantees cleanup even on error.

<Warning>
The agent is instructed to keep variant count under 20. If you extend this recipe, consider adding an explicit guard in the harness to limit concurrent sandboxes.
</Warning>
