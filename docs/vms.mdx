Superserve VMs are Firecracker micro-VMs with hardware-level isolation and sub-200ms boot times. Each VM gets its own kernel, filesystem, and network stack.

## Creating a VM

Specify a name and base image. Optionally configure vCPUs and memory.

<CodeGroup>

```bash CLI
superserve vm create --name my-sandbox --image ubuntu-22.04 --vcpu 2 --memory 512
```

```typescript TypeScript
const vm = await client.vms.create({
  name: "my-sandbox",
  image: "ubuntu-22.04",
  vcpu: 2,
  memory: 512,
})
```

```python Python
vm = client.vms.create(name="my-sandbox", image="ubuntu-22.04", vcpu=2, memory=512)
```

</CodeGroup>

The VM transitions from `CREATING` to `RUNNING` in under 200ms.

## State machine

```
CREATING --> RUNNING --> STOPPED
               |            |
               v            v
           SLEEPING       DEAD
               |
               v
            RUNNING (wake)
```

| State | Description | Transitions |
|-------|-------------|-------------|
| `CREATING` | VM is being provisioned | Automatically moves to `RUNNING` |
| `RUNNING` | VM is active, accepting commands | Can `stop`, `sleep`, or `delete` |
| `STOPPED` | VM process is stopped, disk persists | Can `start` or `delete` |
| `SLEEPING` | VM is suspended to disk with full memory state | Can `wake` or `delete` |
| `DEAD` | VM has been deleted | Terminal state |

### Stopping vs sleeping

- **Stop** shuts down the VM process. The filesystem is preserved, but in-memory state is lost. Starting a stopped VM boots a fresh process with the same disk.
- **Sleep** suspends the entire VM to disk, including memory. Waking a sleeping VM restores the exact process state, open files, and running programs.

Sleep is useful when you want to pause a VM without losing running processes. Stop is useful when you want a clean restart.

## Managing VMs

<CodeGroup>

```bash CLI
superserve vm list
superserve vm list --status RUNNING
superserve vm get vm_a1b2c3d4
superserve vm stop vm_a1b2c3d4
superserve vm start vm_a1b2c3d4
superserve vm sleep vm_a1b2c3d4
superserve vm wake vm_a1b2c3d4
superserve vm delete vm_a1b2c3d4 --force
```

```typescript TypeScript
const vms = await client.vms.list()
const running = await client.vms.list({ status: "RUNNING" })
const vm = await client.vms.get("vm_a1b2c3d4")
await client.vms.stop("vm_a1b2c3d4")
await client.vms.start("vm_a1b2c3d4")
await client.vms.sleep("vm_a1b2c3d4")
await client.vms.wake("vm_a1b2c3d4")
await client.vms.delete("vm_a1b2c3d4")
```

```python Python
vms = client.vms.list()
running = client.vms.list(status="RUNNING")
vm = client.vms.get("vm_a1b2c3d4")
client.vms.stop("vm_a1b2c3d4")
client.vms.start("vm_a1b2c3d4")
client.vms.sleep("vm_a1b2c3d4")
client.vms.wake("vm_a1b2c3d4")
client.vms.delete("vm_a1b2c3d4")
```

</CodeGroup>

## Command execution

Run commands inside a running VM.

<CodeGroup>

```bash CLI
superserve exec vm_a1b2c3d4 "apt-get update && apt-get install -y python3"
superserve exec vm_a1b2c3d4 "python3 /app/script.py" --stream --timeout 60
```

```typescript TypeScript
const result = await client.exec("vm_a1b2c3d4", {
  command: "echo hello world",
})
console.log(result.stdout) // "hello world\n"
```

```python Python
result = client.exec("vm_a1b2c3d4", command="echo hello world")
print(result.stdout)  # "hello world\n"
```

</CodeGroup>

## File operations

Upload files to and download files from a VM.

<CodeGroup>

```bash CLI
superserve files upload vm_a1b2c3d4 ./script.py /app/script.py
superserve files download vm_a1b2c3d4 /app/results.json ./results.json
```

```typescript TypeScript
await client.files.upload("vm_a1b2c3d4", "/app/script.py", buffer)
const content = await client.files.download("vm_a1b2c3d4", "/app/results.json")
```

```python Python
client.files.upload("vm_a1b2c3d4", "/app/script.py", data)
content = client.files.download("vm_a1b2c3d4", "/app/results.json")
```

</CodeGroup>

## SSH access

Open an interactive SSH session to a running VM.

```bash
superserve ssh vm_a1b2c3d4
```

## Use cases

<AccordionGroup>
  <Accordion title="AI Agent Sandboxes">
    Give each AI agent its own isolated VM. The agent can install packages, run code, and modify files without affecting other agents or your infrastructure. Use checkpoints to roll back destructive changes.
  </Accordion>
  <Accordion title="Parallel Experimentation">
    Fork a VM into multiple copies to run experiments in parallel. Each fork is independent -- test different configurations, models, or parameters simultaneously from the same starting point.
  </Accordion>
  <Accordion title="Reproducible Environments">
    Checkpoint a known-good state and roll back to it between test runs. Every run starts from the exact same environment.
  </Accordion>
  <Accordion title="Secure Code Execution">
    Execute untrusted code in hardware-isolated micro-VMs. The code has no access to your infrastructure, other VMs, or the host system.
  </Accordion>
</AccordionGroup>
