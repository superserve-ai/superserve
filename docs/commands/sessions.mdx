`run()` is one-shot: it starts a command and resolves when it finishes. When you need to **interact** with a running process — feed it stdin, signal it, stream output back with low latency — use `spawn()`. It returns a live session handle backed by a WebSocket.

<CodeGroup>
```typescript TypeScript
const session = await sandbox.commands.spawn("python -i", {
  onStdout: (data) => process.stdout.write(data),
})

session.stdin.write("print(2 + 2)\n")
session.stdin.close()

const result = await session.wait()
console.log(`exited ${result.exitCode}`)
```

```python Python
# spawn is async-only — use AsyncSandbox.
session = await sandbox.commands.spawn(
    "python -i",
    on_stdout=lambda data: print(data, end=""),
)

await session.stdin.write("print(2 + 2)\n")
await session.stdin.close()

result = await session.wait()
print(f"exited {result.exit_code}")
```
</CodeGroup>

<Note>
`spawn()` is **async-only in Python** — full-duplex I/O doesn't fit a blocking call, so the synchronous `Sandbox.commands.spawn()` raises and points you to `AsyncSandbox`. TypeScript has a single `spawn()` that works in both Node and the browser.
</Note>

## `run` or `spawn`?

| | `run()` | `spawn()` |
| --- | --- | --- |
| Shape | one-shot, resolves with the result | live handle, returned while running |
| Output | return value (or streaming callbacks) | streaming callbacks + `wait()` |
| Stdin | — | `stdin.write()` / `stdin.close()` |
| Signals | — | `kill(signal)` |
| Transport | HTTP / Server-Sent Events | WebSocket |

Reach for `run()` for scripts and builds. Reach for `spawn()` for REPLs, agents you stream tokens from, or anything you need to talk back to.

## Writing to stdin

`stdin.write()` accepts a string (UTF-8 encoded) or raw bytes, so binary input is sent faithfully. Call `stdin.close()` to signal EOF — many programs (like `cat` or a language REPL) wait for it before finishing.

<CodeGroup>
```typescript TypeScript
const session = await sandbox.commands.spawn("cat", {
  onStdout: (data) => process.stdout.write(data),
})

session.stdin.write("first line\n")
session.stdin.write("second line\n")
session.stdin.close()

await session.wait()
```

```python Python
session = await sandbox.commands.spawn(
    "cat",
    on_stdout=lambda data: print(data, end=""),
)

await session.stdin.write("first line\n")
await session.stdin.write("second line\n")
await session.stdin.close()

await session.wait()
```
</CodeGroup>

## Signals

Send a POSIX signal to the running process — for example, to interrupt or stop it. `kill()` defaults to `SIGTERM`.

<CodeGroup>
```typescript TypeScript
const session = await sandbox.commands.spawn("sleep 100")
session.kill("SIGINT")
const result = await session.wait() // exitCode reflects the signal
```

```python Python
session = await sandbox.commands.spawn("sleep 100")
await session.kill("SIGINT")
result = await session.wait()  # exit_code reflects the signal
```
</CodeGroup>

## Cleanup

Dispose a session to kill the process and close the connection. Use `await using` in TypeScript or `async with` in Python so it happens automatically when the scope exits.

<CodeGroup>
```typescript TypeScript
await using session = await sandbox.commands.spawn("python -i", {
  onStdout: (data) => process.stdout.write(data),
})
session.stdin.write("print('hi')\n")
session.stdin.close()
await session.wait()
// the process is killed and the socket closed here
```

```python Python
async with await sandbox.commands.spawn(
    "python -i",
    on_stdout=lambda data: print(data, end=""),
) as session:
    await session.stdin.write("print('hi')\n")
    await session.stdin.close()
    await session.wait()
# the process is killed and the socket closed here
```
</CodeGroup>

## Network drops

If the connection closes before the command finishes — a transient network hiccup — `wait()` rejects (TypeScript) or raises (Python). The sandbox keeps running the process; reconnect with `Sandbox.connect(id)` if you stored the ID. A paused sandbox is resumed automatically before the session opens.
